TASK 2: EKS Cluster Setup (Run these commands on AWS CLI)
==============================================================
-- Docx Requirements --
  [✓] Create EKS Cluster
  [✓] Configure worker nodes
  [✓] Configure namespaces
  [✓] Configure Kubernetes access control
  [✓] Prepare deployment environment
  [✓] Scalability, Security, HA

-- Commands --
Step 1: Install eksctl (if not installed)
  curl --silent --location "https://github.com/weaveworks/eksctl/releases/latest/download/eksctl_Linux_amd64.tar.gz" | tar xz -C /tmp
  sudo mv /tmp/eksctl /usr/local/bin

Step 2: Create EKS Cluster
  eksctl create cluster \
    --name shine-cluster \
    --region us-east-1 \
    --nodegroup-name shine-nodes \
    --node-type t3.medium \
    --nodes 2 \
    --nodes-min 1 \
    --nodes-max 3 \
    --managed

Step 3: Update kubeconfig (access cluster)
  aws eks update-kubeconfig --name shine-cluster --region us-east-1

Step 4: Create namespaces
  kubectl create namespace retail          ← app goes here
  kubectl create namespace argocd          ← ArgoCD installed here
  kubectl create namespace argo-rollouts   ← Argo Rollouts installed here

Step 5: RBAC - create service account
  kubectl create serviceaccount retail-sa -n retail
  kubectl create clusterrolebinding retail-admin \
    --clusterrole=cluster-admin \
    --serviceaccount=retail:retail-sa

Step 6: Verify cluster
  kubectl get nodes
  kubectl get ns

-- Scalability  : --nodes-min 1 --nodes-max 3 → Cluster Autoscaler handles scale-up/down
-- Security     : IAM roles for node groups, RBAC enabled by default on EKS
-- HA           : EKS managed nodes spread across multiple AZs automatically

  Monitor → New Monitor:

  [✓] Scenario 3: deployment failed

-- Simple AI Agent (Python concept) --

  import anthropic, subprocess

  def collect_logs(pod_name, namespace="retail"):
      # Get pod logs (last 100 lines)
      logs = subprocess.run(
          ["kubectl", "logs", pod_name, "-n", namespace, "--tail=100"],
          capture_output=True, text=True
      ).stdout
      # Get pod events and status
      events = subprocess.run(
          ["kubectl", "describe", "pod", pod_name, "-n", namespace],
          capture_output=True, text=True
      ).stdout
      return logs, events

  def ask_claude(logs, events, scenario):
      client = anthropic.Anthropic(api_key="YOUR_API_KEY")
      msg = client.messages.create(
          model="claude-opus-4-5",
          max_tokens=1024,
          messages=[{
              "role": "user",
              "content": f"Kubernetes issue: {scenario}\n\nLogs:\n{logs}\n\nEvents:\n{events}\n\nFind root cause and suggest fix steps."
          }]
      )
      return msg.content[0].text

  # Run it
  pod = "userprofile-rollout-abc123"
  logs, events = collect_logs(pod)
  print(ask_claude(logs, events, "pod keeps restarting"))

-- Scenario 1: Pod CrashLoopBackOff (continuously restarting) --
  Commands to run first:
    kubectl get pods -n retail
    kubectl describe pod <pod-name> -n retail    ← review pod events
    kubectl logs <pod-name> -n retail --previous ← analyze previous crash logs
  Claude AI analyzes:
    → Missing env var / secret not found → fix: check userprofile-secret.yml
    → OOMKilled → fix: increase memory limits in values.yaml
    → Wrong image → fix: check image tag in values.yaml
    → MongoDB connection refused → fix: check mongodb-deployment is running

-- Scenario 2: Latency increased after deployment --
  Commands to run first:
    kubectl top pods -n retail                   ← analyze performance metrics
    kubectl top nodes                            ← check node pressure
    kubectl logs <pod-name> -n retail            ← review application traces/slow queries
  Claude AI analyzes:
    → MongoDB slow query → fix: add index or increase resources
    → CPU throttling → fix: raise CPU limits in Helm values
    → Memory pressure → fix: check for memory leak in server.js
    → Too many replicas sharing same DB → fix: connection pooling

-- Scenario 3: Deployment failed in Kubernetes --
  Commands to run first:
    kubectl rollout status rollout/userprofile-rollout -n retail  ← analyze deployment status
    kubectl get events -n retail --sort-by=.lastTimestamp         ← review Kubernetes events
    kubectl describe rollout userprofile-rollout -n retail
  Claude AI analyzes:
    → ImagePullBackOff → fix: check Docker Hub credentials, image name in values.yaml
    → Quota exceeded → fix: request higher limits from cluster admin
    → Invalid YAML/config → fix: helm lint ./Retail-chart before deploying
    → Rollback: kubectl argo rollouts undo userprofile-rollout -n retail

-- Tools to evaluate --
  1. Robusta     : Kubernetes-native, auto-collects pod failures, integrates with Claude/ChatGPT for RCA
  2. OpenObserve : Open source Datadog alternative - logs, metrics, traces in one UI
  3. Grafana+Loki: Grafana (dashboards) + Loki (log aggregation) - widely used, free
  4. Keptn       : Event-driven auto-remediation (auto rollback on SLO breach)
  5. Prometheus  : Metrics collection + alerting rules (most popular K8s monitoring)

-- Comparison Table --
  Feature                  | Traditional Monitoring   | AI-Assisted (AIOps)
  -------------------------|--------------------------|--------------------
  Log analysis             | Manual grep/search       | Automatic pattern detection
  Alert noise              | High (many false alarms) | Low (correlated alerts)
  Root cause finding       | Hours of manual work     | Minutes (Claude analyzes instantly)
  Anomaly detection        | Static thresholds only   | ML-based dynamic thresholds
  Incident recommendations | Team experience needed   | AI suggests steps automatically
  MTTR (time to resolve)   | High                     | Significantly reduced

-- Recommended Open Source Stack --
  Prometheus  → metrics scraping from K8s
  Grafana     → dashboards for metrics
  Loki        → log aggregation (like Datadog logs)
  Robusta     → K8s-native alerts + AI-powered RCA
  Claude API  → deep log analysis and remediation advice
